# 实施计划

## 1. 实施目标

把当前项目推进到“可以开始编码且可并行协作”的状态，并尽量降低未来集成返工。

## 2. 集成顺序

推荐按下面顺序推进：

1. 先锁共享契约和数据模型。
2. 再打通提醒调度与通知发送。
3. 接着打通反馈页和反馈提交。
4. 然后接规则引擎和 AI 下一步生成。
5. 最后补日志、晚间收口和验证脚本。

## 3. 并行开发 Lane

### Lane A：共享契约与数据模型

- 目标：定义实体、状态枚举、接口输入输出结构
- 建议分支：`codex/mvp-contracts`
- 依赖：无
- 共享契约：
  - `ReminderPlan`
  - `ReminderRun`
  - `FeedbackEvent`
  - `ActionPlan`
  - `DailyClosure`
- 预期产出：
  - 数据库 schema 初稿
  - API DTO 定义
  - 事件状态流转图

### Lane B：提醒调度与通知

- 目标：跑通到点触发、记录运行、发送消息
- 建议分支：`codex/mvp-scheduler-notify`
- 依赖：Lane A
- 共享契约：
  - 到期扫描接口
  - 通知发送结果结构
  - `ReminderRun.status`
- 预期产出：
  - 调度器 job
  - 通知适配器
  - 推送失败重试逻辑

### Lane C：反馈页与快捷交互

- 目标：构建 PWA 反馈页和快捷按钮交互
- 建议分支：`codex/mvp-feedback-web`
- 依赖：Lane A
- 共享契约：
  - `GET /r/{runToken}`
  - `POST /api/runs/{runId}/feedback`
- 预期产出：
  - 反馈页 UI
  - 提交反馈流程
  - 完成/跳过/滑坡等状态展示

### Lane D：规则引擎与 AI 规划

- 目标：根据反馈输出下一步动作，并在 AI 不可用时兜底
- 建议分支：`codex/mvp-rules-llm`
- 依赖：Lane A，建议在 Lane B、C 契约稳定后接入
- 共享契约：
  - `feedback_type`
  - `intensity_level`
  - `ActionPlan`
- 预期产出：
  - 三档动作决策逻辑
  - 滑坡恢复协议
  - AI Prompt 和回退策略

### Lane E：日志、收口和验证

- 目标：沉淀执行记录、夜间收口和手动验收能力
- 建议分支：`codex/mvp-validation`
- 依赖：Lane B、C、D
- 共享契约：
  - 执行日志查询接口
  - `DailyClosure`
  - 测试样例输入输出
- 预期产出：
  - 日志页面或日志接口
  - 夜间收口生成器
  - 手动验收脚本和测试清单

## 4. Merge 风险

### 4.1 最大风险点

- Lane A 未先稳定，导致后续接口反复改动
- 反馈类型和动作等级命名不统一
- 前端直接依赖未定稿的 LLM 输出文案格式
- 通知链接 token 机制后补，造成链路重写

### 4.2 控制方式

- 先冻结共享枚举和 DTO
- 前后端都只依赖结构化 `ActionPlan`
- LLM 输出先映射到统一 schema，再给前端消费
- 任何新增反馈类型都先改契约，再改业务逻辑

## 5. Review Gate

建议保留这 4 个 review gate：

1. 需求基线 review
2. 契约与架构 review
3. 伪代码与规则 review
4. 编码后自用链路 review

## 6. 下一步最值得开始的工作

如果下一轮开始真正写代码，最优先的是：

1. 选定最终技术栈并创建目录结构。
2. 定义共享 schema。
3. 实现最小提醒运行链：固定提醒 -> 发送消息 -> 打开反馈页。
