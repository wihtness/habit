# 架构说明

## 1. 架构目标

MVP 架构的目标不是追求平台完备，而是用最小复杂度跑通“日类型判定 -> 模板选取 -> 提醒 -> 反馈 -> 下一步动作 -> 结果记录 -> 晚间收口”这条主链路。

在没有额外约束时，默认推荐技术组合：

- 前端：React + Vite + PWA
- 后端：FastAPI
- 定时任务：APScheduler
- 数据库：SQLite
- 推送：通知适配层，MVP 首接 Bark
- AI：大模型 API，失败时回退到规则模板

## 2. 组件划分

### 2.1 Web/PWA 反馈端

职责：

- 展示提醒详情和当前建议动作
- 提供一键反馈按钮
- 展示 AI 生成的下一步安排
- 展示晚间收口结果和执行日志摘要

边界：

- 不负责调度提醒
- 不内置复杂任务系统
- 不承担长期离线计算

### 2.2 API 服务

职责：

- 提供日程模板、单天覆盖、提醒、反馈、计划、日志和收口相关接口
- 统一组织规则引擎和 LLM 调用
- 管理状态流转和结果落库

### 2.3 调度器

职责：

- 在每天开始时或首次运行时解析当天日类型
- 选中日程模板并生成当天提醒计划
- 按计划触发提醒
- 创建本次提醒运行记录
- 调用通知服务发送消息
- 在等待窗口后检查是否无回复，并触发默认动作

### 2.4 通知适配层

职责：

- 抽象 Bark、Web Push、Telegram 等不同推送渠道
- 对外暴露统一发送接口
- 记录发送结果、失败原因和重试次数

MVP 约束：

- 只实现一个主渠道
- 渠道配置保存在本地配置表中

### 2.5 规则引擎

职责：

- 根据提醒类型、反馈类型、身体状态、时间段，生成正常版、降级版或保底版动作
- 在 LLM 不可用或超时时输出兜底动作

### 2.6 LLM 规划器

职责：

- 接收上下文并生成下一步 20 到 60 分钟动作
- 严格遵守“不扩展任务、不启动新项目、必须有停止标准”的约束

### 2.7 日类型与模板解析器

职责：

- 判断某一天属于工作日、普通休息日还是节假日
- 按优先级选择对应模板
- 处理用户的单天覆盖设置
- 在模板缺失时提供明确降级策略
- 读取本地节假日日期配置

## 3. 领域对象

### 3.1 ScheduleTemplate

- 用途：定义某类日子下的提醒模板集合
- 关键字段：`id`、`name`、`day_type`、`description`、`is_enabled`

### 3.2 ScheduleTemplateItem

- 用途：定义模板内的单条提醒项
- 关键字段：`id`、`template_id`、`time`、`type`、`goal_context`、`default_action_level`、`feedback_window_minutes`

### 3.3 CalendarDayOverride

- 用途：对单天的日类型做手动覆盖
- 关键字段：`id`、`date`、`day_type`、`reason`

### 3.4 ReminderPlan

- 用途：由模板为某一天物化出来的提醒计划
- 关键字段：`id`、`template_item_id`、`plan_date`、`name`、`time`、`type`、`goal_context`

### 3.5 ReminderRun

- 用途：记录某次实际触发的提醒
- 关键字段：`id`、`plan_id`、`scheduled_at`、`triggered_at`、`status`、`notification_channel`、`feedback_deadline`

### 3.6 FeedbackEvent

- 用途：记录用户点击的反馈和备注
- 关键字段：`id`、`run_id`、`feedback_type`、`note`、`created_at`

### 3.7 ActionPlan

- 用途：记录系统最终给出的动作方案
- 关键字段：`id`、`run_id`、`source`、`intensity_level`、`plan_text`、`duration_minutes`、`stop_condition`

### 3.8 HealthRule

- 用途：定义某类身体状态的危险信号、建议动作和禁忌动作
- 关键字段：`id`、`symptom_type`、`danger_signals`、`recommended_actions`、`avoid_actions`

### 3.9 DailyClosure

- 用途：记录晚间收口总结
- 关键字段：`id`、`date`、`summary_text`、`tomorrow_first_step`、`created_at`

## 4. 关键接口契约

### 4.1 调度触发

- 内部调用：`prepare_day_schedule(date)`、`run_due_reminders(now)`
- 输入：当前日期、当前时间
- 输出：新建的 `ReminderRun` 列表和发送结果

### 4.2 日类型解析

- 内部调用：`resolve_day_type(date)`
- 输入：日期、周几、本地节假日配置、单天覆盖配置
- 输出：`workday | weekend | holiday`

### 4.3 获取提醒反馈页

- `GET /r/{runToken}`
- 输入：提醒运行令牌
- 输出：提醒内容、快捷反馈按钮、当前默认动作、上下文摘要

### 4.4 提交反馈

- `POST /api/runs/{runId}/feedback`
- 输入：`feedback_type`、可选备注、可选身体状态
- 输出：最新 `ActionPlan`

### 4.5 获取下一步动作

- `POST /api/runs/{runId}/next-step`
- 输入：反馈事件、运行上下文、健康状态、目标上下文
- 输出：规则结果或 LLM 生成结果

### 4.6 晚间收口

- `POST /api/daily-closure/generate`
- 输入：某天的提醒记录、反馈记录、执行结果
- 输出：收口文本、明日第一步

## 5. 数据流主链

1. 日类型解析器判断当天是 `workday`、`weekend` 还是 `holiday`。
2. 系统选择对应 `ScheduleTemplate`。
3. 系统将模板物化为当天的 `ReminderPlan`。
4. 调度器扫描到期 `ReminderPlan`。
5. 系统创建 `ReminderRun`。
6. 通知适配层发送推送，消息内带反馈页链接。
7. 用户点击链接进入反馈页。
8. 用户点击反馈，生成 `FeedbackEvent`。
9. API 先走规则引擎，再按需要调用 LLM。
10. 系统生成并保存 `ActionPlan`。
11. 用户后续可标记完成、跳过或再次求助。
12. 晚间任务汇总当日记录，生成 `DailyClosure`。

## 6. 失败点与降级策略

### 6.1 推送失败

- 处理：记录失败状态，按渠道配置重试一次
- 降级：在管理页显示未送达提醒，供手动补看

### 6.2 用户无回复

- 处理：等待窗口结束后直接生成默认最低动作
- 降级：标记为 `no_response_auto_progress`

### 6.3 LLM 超时或不可用

- 处理：回退到规则引擎模板
- 要求：用户仍然能拿到可执行动作

### 6.4 SQLite 锁冲突或写入失败

- 处理：关键事件先内存排队后重试
- 要求：提醒触发和反馈记录不能静默丢失

### 6.5 节假日配置缺失或模板缺失

- 处理：优先记录配置缺口并回退到工作日模板
- 要求：不能因为模板缺失让当天完全没有提醒链

## 7. 代码组织建议

如果下一步按默认技术栈落地，建议采用如下结构：

```text
apps/
  api/
  web/
packages/
  shared/
docs/
```

其中：

- `apps/api` 放 FastAPI、调度器、规则引擎、通知适配器
- `apps/api` 还应包含日类型解析器和模板物化逻辑
- `apps/web` 放 PWA 反馈页和管理页
- `packages/shared` 放共享类型、枚举和接口约束

## 8. Review Checkpoints

在进入业务代码前，建议按下面顺序过 review：

1. 共享数据模型和状态枚举
2. 日类型与模板选择契约
3. 提醒运行与反馈接口契约
4. 规则引擎输出结构
5. LLM Prompt 约束和回退策略
